## V1.3 — 智能重要度排序 + 按筛选独立 AI 分析

**日期**：2026-03-31 ~ 2026-04-01  
**状态**：已完成

### 背景

V1.2 完成内容分类后，两个核心功能仍有缺陷：

1. **"重要程度"排序是假的**：仅按 `score DESC, commentsCount DESC` 三重排序，不同数据源的 score 量纲完全不同（Twitter ~20K-46K，Bilibili 视频 ~百万播放，GitHub ~几百 stars，HN ~几百分），导致 Twitter/Bilibili 永远排在最前面，跨源比较无意义。
2. **AI 分析两个 Tab 完全相同**：`loadAnalysis()` 获取全局唯一分析结果后同时渲染到 "AI 热点" 和 "综合热点" 两个面板，内容完全一致，没有针对各自板块和筛选条件做差异化分析。

### 目标

- 重要度排序：跨数据源可比较，综合热度、互动、多源交叉出现程度
- AI 分析：按当前 Tab 的 category + 所有筛选条件独立生成/缓存，筛选变化后自动触发（5s 防抖）

### 技术方案

#### 1. 重要度评分引擎

新建 `src/services/importance.ts`，采用**三因子组合评分**模型：

| 因子 | 权重 | 计算方式 |
|------|------|---------|
| 归一化热度 | 40% | 同源内 score 按百分位排名 → 0~100 |
| 归一化互动 | 30% | 同源内 commentsCount 按百分位排名 → 0~100 |
| 跨源频率 | 30% | 标题 token 倒排索引，检测多源重复话题 → 每匹配 1 源 +25，上限 100 |

**公式**：`importanceScore = normalizedScore × 0.4 + normalizedComments × 0.3 + crossSourceBonus × 0.3`

**标题 Tokenize 策略**：
- 英文：提取 ≥3 字符的单词，过滤 60+ 停用词（the, and, for…）
- 中文/日韩：CJK 字符串按 bigram 切分（"人工智能" → "人工", "工智", "智能"）
- 混合文本同时处理两种

**百分位归一化**：同源内按原始值排序，`percentile = rank / (n-1) × 100`，消除跨源量纲差异。

**评分触发时机**：
- 每次定时采集完成后自动计算
- 手动刷新后自动计算
- 首次部署 backfill 全量历史数据

#### 2. 数据层变更

```prisma
model TrendItem {
  // ... 现有字段
  importanceScore Float @default(0)       // 新增：预计算重要度分数 0~100
  @@index([importanceScore])              // 新增索引
}

model TrendAnalysis {
  // ... 现有字段
  category   String @default("all")       // 新增：分析所属分类
  filterHash String @default("")          // 新增：筛选条件 MD5 哈希（前 12 位）
  @@index([category, filterHash])         // 新增复合索引
}
```

#### 3. AI 分析按筛选条件独立

**后端**（`src/services/analysis.ts`）：

```typescript
interface AnalysisFilter {
  category?: string   // 'ai' | 'general' | 'all'
  source?: string     // 逗号分隔多源
  search?: string     // 搜索关键词
  days?: number       // 时间范围
  minScore?: number   // 热度门槛
}
```

- `analyzeTrends(filter)` 按筛选条件构建 Prisma where → 取 Top 80 条 → 发给 OpenRouter 分析
- Prompt 中注入分类标签（"AI/科技" / "综合" / "全部"），让 AI 理解分析范围
- `buildFilterHash(filter)` 将筛选条件序列化后取 MD5 前 12 位作缓存 key
- 缓存查询：`getLatestAnalysis(category, filterHash)` → 按 category + filterHash 查最新
- 过期策略：前端检测分析时间 > 1 小时则自动触发刷新

**API 变更**：
- `GET /api/trends/analysis?category=ai&source=reddit&days=7&...` → 返回匹配筛选条件的缓存分析
- `POST /api/trends/analyze` → body 中传入 `{ category, source, search, days, minScore }`

**前端**（`public/app.js`）：
- 新增 `scheduleAnalysis(category, state)` — 5s 防抖定时器
- `reloadAi()` / `reloadGeneral()` 每次调用后自动 schedule 对应 tab 的分析
- `loadAnalysisForTab(category, state)` — 按当前筛选条件请求缓存分析
- Socket `analysis-update` 按 `data.category` 路由到对应 tab 面板
- 手动 "AI 分析" 按钮自动识别当前 tab 发对应请求
- 分析加载中显示 "正在分析中…" 状态

#### 4. 排序逻辑变更

```diff
  case 'importance':
-   orderBy = [{ score: 'desc' }, { commentsCount: 'desc' }]
-   useRawImportance = true
+   orderBy = [{ importanceScore: 'desc' }, { fetchedAt: 'desc' }]
```

删除 `useRawImportance` 分支，统一走 Prisma `orderBy`。

#### 5. 定时任务变更

```
采集完成 → 计算重要度 → 分析 AI 分类 → 分析综合分类
```

Scheduler 在 `collectAll()` 后依次调用 `importance.computeScores()` 和两次 `analysis.analyzeTrends({ category })` → 通过 Socket 推送到各自 tab。

## V1.2 — 基于内容的智能分类系统

**日期**：2026-03-31  
**状态**：已完成

### 背景

V1.1 中 "AI 热点" 和 "综合热点" 的划分方式是**按数据源硬编码**（GitHub/HF/HN/Twitter/BingNews → AI，Google/Reddit/DDG/V2EX/Bilibili → 综合）。问题：

- Reddit 上的 AI 帖子（如 r/LocalLLaMA）被归到综合板块
- Bilibili 的 "谷歌AI内存压缩算法遭质疑" 热搜被归到综合板块
- Google Trends 偶尔出现 AI 相关趋势也无法进入 AI 板块
- 分类逻辑与内容无关，仅靠来源区分，不符合实际需求

### 目标

- 所有数据源的内容按**语义**分类，AI 相关内容无论来源都归入 "AI 热点"
- V1 固定 `ai` / `general` 两个板块
- 架构预留 V2 用户自定义板块（如 "区块链"、"游戏"、"政治"）

### 技术方案

#### 1. 数据层变更

```prisma
model TrendItem {
  // ... 现有字段
  category    String   @default("general")  // 新增：内容分类
  // ...
  @@index([category])                       // 新增索引
}

// V2 预留：分类规则表
model CategoryRule {
  id          Int      @id @default(autoincrement())
  name        String   @unique              // 板块名称，如 "ai", "blockchain"
  label       String                        // 显示名称，如 "AI 热点"
  keywords    String                        // JSON 数组：匹配关键词
  description String?                       // 板块描述
  isSystem    Boolean  @default(false)      // true=系统内置, false=用户自建
  priority    Int      @default(0)          // 匹配优先级（高优先）
  createdAt   DateTime @default(now())
}
```

#### 2. 分类引擎（混合模式）

新建 `src/services/classifier.ts`，两阶段分类：

```
采集完成 → Phase A: 关键词规则匹配 → Phase B: AI 模型判断 → 入库
              ↓ 命中                    ↓ 未命中
           category='ai'           发给 OpenRouter 批量判断
                                       ↓ API 不可用
                                   降级为 'general'
```

**Phase A — 关键词规则匹配**（零成本、零延迟）
- 扫描 `title` + `extra` 中的描述/摘要字段
- 预置 AI 关键词库（约 60+ 词），大小写不敏感：

| 类别 | 关键词示例 |
|------|-----------|
| 通用 | AI, artificial intelligence, 人工智能, machine learning, 机器学习, deep learning, 深度学习 |
| 模型 | LLM, GPT, Claude, Gemini, DeepSeek, 千问, Qwen, Llama, Mistral, BERT, transformer |
| 产品 | OpenAI, Anthropic, HuggingFace, Copilot, ChatGPT, Midjourney, Stable Diffusion, Sora |
| 技术 | AIGC, AGI, RAG, fine-tune, RLHF, LoRA, embedding, inference, tokenizer, neural network |
| 领域 | NLP, computer vision, 自动驾驶, 大模型, 开源模型, prompt engineering, AI agent |

- 命中任意关键词 → `category = 'ai'`，跳过 Phase B
- 匹配逻辑：单词边界匹配（英文）+ 包含匹配（中文），避免 "MAIL" 误命中 "AI"

**Phase B — AI 大模型批量判断**（仅处理 Phase A 未分类项）
- 将未分类项的标题批量发送给 OpenRouter（每批 ≤50 条）
- Prompt：返回 JSON 数组标注每条是 `"ai"` 还是 `"general"`
- 降级策略：API 超时/不可用 → 全部标记为 `"general"`
- 使用现有的 `OPENROUTER_API_KEY` 和 model 配置

#### 3. 采集流程变更

```
Before:  fetch all sources → save to DB
After:   fetch all sources → classifier.classify(items) → save to DB (with category)
```

- `collector.ts` 的 `saveItems()` upsert 中 create/update 都写入 `category`
- 分类在内存中完成，不额外查 DB

#### 4. API 层变更

```diff
// src/routes/trends.ts
- const AI_SOURCES = ['github', 'huggingface', 'hackernews', 'twitter', 'bingnews']
- const GENERAL_SOURCES = ['google', 'reddit', 'duckduckgo', 'v2ex', 'bilibili']
- if (category === 'ai') where.source = { in: AI_SOURCES }
+ if (category) where.category = category
```

- `GET /api/trends?category=ai` → 查 DB 的 `category` 字段
- Source filter 不再受板块限制，任一板块可出现任一来源

#### 5. 前端变更

- 删除 `loadSources()` 中的 `aiSources` / `generalSources` 硬编码过滤
- 两个 tab 的 source filter 显示该分类中**实际存在**的数据源
- 数据渲染逻辑不变

### 涉及文件

| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `prisma/schema.prisma` | 修改 | TrendItem 加 category 字段 + CategoryRule model |
| `src/services/classifier.ts` | **新建** | 混合分类引擎 |
| `src/services/collector.ts` | 修改 | collectAll 集成 classifier，saveItems 写入 category |
| `src/routes/trends.ts` | 修改 | 删除硬编码源数组，改查 category 字段 |
| `public/app.js` | 修改 | 删除 source-based 板块过滤 |

### V2 扩展点

- `CategoryRule` 开放前端管理 UI，用户可创建自定义板块
- `classifier.ts` 从 DB 加载 `CategoryRule.keywords`，支持多板块
- 新增 `GET /api/trends/categories` 返回板块列表，前端动态渲染 tab
- 支持一条内容归属多个板块（多对多关系）

## V1.1 — 信息源扩展 + 质量优化

**日期**：2026-03-28 ~ 2026-03-29  
**状态**：已完成

### 背景

V1.0 完成基础功能后存在以下问题：
- DuckDuckGo 搜索结果包含大量 Wikipedia 垃圾内容
- Reddit 抓取 r/popular 导致内容与 AI/技术无关
- 无中文信息源，缺少 DeepSeek、千问等国内 AI 覆盖
- Google Trends 链接全部指向 RSS feed URL，无法打开
- Twitter API 频繁 429 限流
- 缺少 publishedAt 时间字段，无法区分新旧内容

### 变更内容

#### 新增 5 个数据源

| 数据源 | 文件 | 采集方式 | 内容类型 |
|--------|------|---------|---------|
| GitHub Trending | `src/sources/github.ts` | HTML 爬取 github.com/trending | 每日热门仓库（stars、语言） |
| HuggingFace | `src/sources/huggingface.ts` | REST API | 热门模型 + 每日论文 |
| V2EX | `src/sources/v2ex.ts` | JSON API | 中文技术社区热帖 |
| Bing News | `src/sources/bingnews.ts` | RSS 爬取 | AI 相关新闻（含 DeepSeek/Qwen） |
| Bilibili | `src/sources/bilibili.ts` | JSON API | B站热搜 + 科技区排行 |

#### 已有数据源优化

| 数据源 | 优化内容 |

| HackerNews | 从 Firebase topstories（30 次请求）切换到 Algolia API（单次请求）；增加 points≥20 过滤 |

#### 数据模型变更


#### 前端变更

- 各数据源专属元数据展示（GitHub ⭐/语言、HF 模型类型/likes、B站 播放/弹幕等）

#### Collector 变更

- `saveItems()` upsert 增加 `url` 到 update 块（修复旧数据 URL 不更新问题）
- 注册 10 个数据源（8 固定 + Twitter 按 KEY 条件注册）

## V1.0 — 基础系统搭建

**日期**：2026-03-27  
**状态**：已完成

### 功能

- Express 5 + Prisma ORM + SQLite 后端
- 5 个初始数据源：Google Trends、Reddit、HackerNews、DuckDuckGo、Twitter/X
- Socket.IO 实时推送
- node-cron 每 30 分钟自动采集
- 关键词监控模块（添加关键词 → 自动检查 → 告警推送）
- AI 趋势分析（OpenRouter API → Gemini 模型）
- 前端 SPA：热点列表 + 来源筛选 + 分页 + 手动刷新 + AI 分析面板

### 架构

```
Frontend (HTML/CSS/JS + Socket.IO)
  ↕ HTTP REST + WebSocket
Express 5 Server
  ├── API Routes (trends, keywords)
  ├── Socket.IO (real-time push)
  ├── Cron Scheduler (30min cycle)
  ├── Source Collector (5 sources)
  ├── Keyword Monitor
  ├── Analysis Service (OpenRouter)
  └── Prisma ORM → SQLite
```

### 数据模型

- `TrendItem` — 热点条目（title, url, source, score, extra, externalId）
- `FetchLog` — 采集日志
- `TrendAnalysis` — AI 分析结果
- `MonitorKeyword` — 监控关键词
- `KeywordAlert` — 关键词告警
- `KeywordTrend` — 关键词热度趋势

### 技术栈

| 模块 | 选型 |
|------|------|
| 后端 | Express 5.x + TypeScript |
| 数据库 | Prisma ORM + SQLite |
| 实时通信 | Socket.IO 4.x |
| 定时任务 | node-cron (CronJob) |
| 爬虫解析 | cheerio + 原生 fetch |
| AI 分析 | OpenRouter API (Gemini) |
| Twitter | twitterapi.io (REST) |
| 开发工具 | tsx watch 热重载 |
